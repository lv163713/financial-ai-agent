# 金融资讯智能分析系统 - 开发文档（RAG优先查询工作流）

## 1. 文档目标
本文档用于明确下一阶段开发方向：在现有抓取与分析能力基础上，构建“每日定时更新 + 用户自由查询 + RAG优先检索 + 实时爬取回退”的完整工作流。

## 2. 目标能力与范围

### 2.1 业务目标
- 每日自动产出可用财经资讯分析（固定时间执行）。
- 支持用户输入自然语言问题，自由指定想关注的财经内容与板块。
- 在保证响应速度的同时，尽可能返回近期（默认 24h/48h）高相关资讯分析结果。

### 2.2 本阶段范围（In Scope）
- 每日 08:00 定时抓取与分析。
- 用户查询协议设计（关键词、板块、市场、时间范围）。
- AI 意图解析与结构化参数提取。
- RAG 向量检索优先，未命中时触发实时爬取。
- 二次分析输出统一结构化结果。

### 2.3 非本阶段范围（Out of Scope）
- 多模型路由与自动评测平台。
- 复杂量化回测与交易信号自动执行。
- 多租户权限体系与计费能力。

## 3. 总体方案

### 3.1 双通道架构
- 通道 A（定时通道）：每日 08:00 自动抓取核心数据源，完成分析并写入数据库与向量库。
- 通道 B（用户通道）：用户发起查询，优先走向量检索；若检索不足，触发实时抓取并回填分析。

### 3.2 核心原则
- 时效优先：默认只看最近 24h，可扩展到 48h。
- 速度优先：先查 RAG，减少不必要抓取与模型调用。
- 质量优先：检索命中不足时自动回退抓取，避免“无内容硬回答”。
- 可追踪：每次查询与抓取都保留审计信息（成功、失败、重试次数、来源）。

## 4. 查询协议（用户输入协议）

### 4.1 用户输入（自然语言）
示例：
- “最近 24 小时新能源板块有什么利空消息？”
- “美股半导体最近两天有哪些重要事件？”

### 4.2 AI 结构化输出字段
- `raw_query`: 用户原始问题
- `keywords`: 关键词列表
- `sector`: 板块（可为空）
- `market`: 市场范围（A股/美股/港股/全球）
- `time_range_hours`: 时间范围（默认 24，可选 48）
- `intent_type`: 查询类型（概览/风险/机会/单主题追踪）

### 4.3 协议约束
- 保留 `raw_query` 作为硬约束，防止意图漂移。
- 若 AI 未识别到 `sector`，允许按 `keywords` 与 `market` 检索。
- 超出范围参数按默认值回退（如时间范围回退到 24h）。

## 5. RAG优先工作流

### 5.1 流程步骤
1. 接收用户输入并完成意图结构化。
2. 按结构化条件检索向量库（先 24h，再按配置放宽到 48h）。
3. 判断命中质量：
   - 相关度达到阈值且时间窗满足，进入分析阶段。
   - 相关度不足或时效不足，触发实时抓取。
4. 实时抓取后执行 AI 分析，并写回数据库与向量库。
5. 汇总“历史命中 + 新抓取”结果，统一生成回答。

### 5.2 命中判定建议
- `similarity_threshold`: 建议 0.75（可配置）
- `min_docs`: 最小有效文档数建议 3
- `freshness_window`: 默认 24h，允许 48h 兜底
- `source_diversity`: 至少 2 个来源优先

### 5.3 回退抓取策略
- 当 `min_docs` 不满足或时效不满足时触发抓取。
- 抓取支持批量与重试（已具备基础能力）。
- 抓取结果必须回灌向量库，形成“越查越全”的闭环。

## 6. 定时任务方案（每日 08:00）

### 6.1 调度目标
- 每日 08:00 自动执行一次“核心源批量抓取 + 分析 + 入库 + 入向量库”。

### 6.2 任务结构
- 任务输入：预定义财经源清单（可按市场/板块分类）。
- 任务执行：批量抓取 -> 分析 -> 存储 -> 向量化写入。
- 任务输出：任务汇总（总数、成功数、失败数、失败原因）。

### 6.3 失败处理
- 单条失败不影响整批。
- 按配置重试（建议 1~2 次）。
- 记录审计日志用于追踪与告警。

## 7. 接口契约草案

### 7.1 用户查询接口（新增）
- `POST /query/analyze`
- 请求体：
  - `query`: string
  - `time_range_hours`: int（可选，默认 24）
  - `market`: string（可选）
- 返回体：
  - `intent`: 结构化意图
  - `evidence`: 命中文档与来源列表
  - `analysis`: 结论、影响评估、受影响板块、逻辑推演
  - `meta`: 检索命中数、是否触发实时抓取、处理耗时

### 7.2 每日任务接口（可选）
- `POST /jobs/daily-run`（手动触发补跑）
- `GET /jobs/daily-run/{job_id}`（查看执行结果）

## 8. 数据与存储设计

### 8.1 结构化数据（MySQL）
- 现有 `news`、`analysis` 继续作为主业务表。
- 现有审计日志表继续记录批量执行与重试情况。

### 8.2 向量库数据
- 文档主键：`news_id` + `chunk_id`
- 元数据建议包含：
  - `source`
  - `published_at`
  - `market`
  - `sectors`
  - `ingest_time`
- 检索过滤优先使用 `published_at` 与 `market`。

## 9. 输出格式规范（面向前端）
- `summary`: 一句话结论
- `impact_assessment`: 利好/利空/中性
- `affected_sectors`: 板块列表
- `logical_reasoning`: 逻辑推演
- `evidence`: 来源、URL、发布时间、相关度
- `risk_notice`: 不确定性提示（数据不足/来源单一/时效边界）

## 10. 验收标准

### 10.1 功能验收
- 每日 08:00 定时任务可稳定执行。
- 用户输入任意财经问题可得到结构化回答。
- RAG 命中不足时可自动触发实时抓取并完成分析。

### 10.2 质量验收
- 默认 24h 时间窗下，查询结果具备有效来源与时间戳。
- 关键字段完整率达到 95% 以上。
- 查询接口在命中 RAG 场景下具备明显更低延迟。

### 10.3 可运维验收
- 可查询任务成功率、失败率、重试次数。
- 失败原因可追踪到具体 URL 与步骤。

## 11. 迭代里程碑建议
- M1：完成查询协议与 AI 意图结构化。
- M2：打通 RAG 优先 + 回退抓取闭环。
- M3：接入每日 08:00 调度与任务看板。
- M4：前端查询页联调与验收上线。

## 12. 风险与对策
- 风险：匿名抓取受风控导致命中不足。
  - 对策：增强多源策略与鉴权抓取方案。
- 风险：AI 意图解析偏差导致检索偏离。
  - 对策：保留 `raw_query` 作为硬约束，加入规则校正。
- 风险：检索到旧闻影响判断。
  - 对策：强制时间窗过滤，超时自动降权或剔除。
